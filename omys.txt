당신은 풀스택 개발자입니다. 실제 사용자에게 공개할 수 있는 모바일 웹 MVP인
“OMYS(오늘의 미스터리 스팟)”를 구현해 주세요.

단순한 UI 프로토타입이 아니라, 여러 사용자가 초대 링크로 접속하여 실제로
미스터리 스팟을 정하고 사용할 수 있는 완성된 MVP가 목표입니다.

먼저 현재 저장소의 파일 구조, 기존 코드, 사용 기술, 실행 방법을 확인하세요.

- 기존 프로젝트가 있다면 기존 구조와 기술을 최대한 유지하세요.
- 빈 프로젝트라면 아래 기본 기술 스택을 사용하세요.
- 구현 전에 간단한 작업 계획과 데이터 구조를 제시한 뒤 바로 개발하세요.
- 작업 도중 필요한 패키지와 API를 직접 확인하고 설치하세요.
- 중요하지 않은 사항은 합리적으로 결정하고 계속 진행하세요.
- 외부 API 키처럼 반드시 사용자 입력이 필요한 경우에만 질문하세요.
- 모든 핵심 기능을 구현하고 실제로 실행·테스트한 뒤 결과를 보고하세요.

# 1. 서비스 개요

서비스명: OMYS(오늘의 미스터리 스팟)

OMYS는 친구나 연인이 무엇을 할지 결정하기 어려울 때,
목적지를 무작위로 정하고 도착할 때까지 목적지를 숨겨 주는 서비스입니다.

핵심 경험은 다음과 같습니다.

1. 조건 또는 후보 장소 입력
2. 미스터리 스팟 무작위 선정
3. 선정 결과 잠금
4. 목적지를 숨긴 상태로 이동
5. 도착 직전 또는 도착 시 목적지 공개
6. 결과 공유

모바일 사용을 우선하여 반응형 웹으로 구현하세요.

# 2. 기본 기술 스택

기존 프로젝트가 없다면 다음 구성을 사용하세요.

프런트엔드:
- React
- TypeScript
- Vite
- 모바일 우선 반응형 UI
- 지도 SDK

백엔드:
- FastAPI
- SQLAlchemy
- Pydantic
- REST API
- 개발 환경 SQLite
- 운영 환경에서는 PostgreSQL로 교체할 수 있는 구조

실시간 동기화:
- MVP에서는 2~3초 간격 polling 사용
- 구조가 복잡해지지 않는다면 WebSocket 사용 가능

로그인은 MVP에서 필수가 아닙니다.
초대받은 사용자가 회원가입 때문에 이탈하지 않도록 닉네임과 방 링크만으로
참여할 수 있게 구현하세요.

# 3. 핵심 모드

OMYS에는 두 가지 모드가 있습니다.

## A. 친구들과 미스터리 스팟 선정

### 사용자 흐름

1. 방장이 방을 생성한다.
2. 다음 정보를 설정한다.
   - 방 이름
   - 출발 위치
   - 참가 마감 여부
   - 다시 뽑기 허용 여부
3. 초대 링크가 생성된다.
4. 친구들이 링크로 들어와 닉네임을 입력한다.
5. 각자 가고 싶은 장소를 하나 이상 비밀리에 제출한다.
6. 모든 참가자가 제출을 완료하면 방장이 추첨을 시작한다.
7. 제출된 후보 중 하나를 서버에서 무작위로 선정한다.
8. 선정 결과는 잠기며 임의로 바뀌지 않는다.
9. 당첨 장소를 제출한 사람만 목적지를 확인할 수 있다.
10. 나머지 참가자는 도착할 때까지 목적지를 알 수 없다.
11. 도착하면 모든 참가자에게 장소를 공개한다.

### 장소 제출

사용자는 다음 두 가지 방법으로 장소를 제출할 수 있어야 합니다.

1. 장소 직접 검색
2. 카테고리에서 종목을 고른 뒤 출발 위치 주변 검색 결과에서 선택

직접 입력한 문자열만 저장하지 말고 가능하면 장소 검색 API 결과에서
선택하도록 하세요. 다음 정보를 저장합니다.

- 외부 장소 ID
- 장소명
- 카테고리
- 주소
- 위도
- 경도
- 예상 가격대
- 영업 여부
- 장소 URL

### 추천 카테고리

최소한 다음 카테고리를 제공하세요. 카테고리를 선택하면 실제 장소 대신
세부 종목을 먼저 보여 줍니다. 추천 종목을 선택하거나 원하는 종목을 직접 입력한 뒤
확인 버튼을 눌렀을 때 출발 위치 주변 장소를 검색합니다.

- 게임·실내 놀거리
- 운동·액티비티
- 관광·산책
- 쇼핑·구경
- 데이트코스·이색 체험

예를 들어 `게임·실내 놀거리`에서는 보드게임, 피씨방, 플스방, 방탈출 등의
종목을 제공합니다. 맛집과 디저트 카테고리는 제공하지 않습니다.

출발 위치를 기준으로 가까운 장소를 보여 주세요.
추천 결과에는 장소명, 카테고리, 대략적인 거리, 영업 여부만 표시하세요.

다른 참가자가 제출한 장소는 추첨 전까지 볼 수 없어야 합니다.
참가자 수와 제출 완료 여부만 보여 주세요.

## B. OMYS 자체 미스터리 스팟 선정

사용자가 직접 장소를 제출하지 않고 조건만 입력하면,
OMYS가 조건에 맞는 장소 후보를 찾은 뒤 그중 하나를 무작위로 선정합니다.

### 입력 조건

- 현재 위치 또는 출발 위치
- 이동 수단
  - 도보
  - 대중교통
  - 자동차
- 최대 이동 가능 시간
- 1인당 예산
- 참가 인원
- 선호 성향 또는 카테고리
- 실내·야외 선호
- 제외할 활동
- 음식 포함 여부
- 접근성 관련 조건
- 이용 가능한 총시간

모든 조건을 필수로 만들지 말고,
위치와 최대 이동 시간 정도만 필수로 설정하세요.

### 선정 방식

1. 출발 위치 주변의 실제 장소를 검색한다.
2. 제외 조건에 해당하는 장소를 제거한다.
3. 현재 영업 중이 아닌 장소를 제거한다.
4. 이동 가능 시간을 초과하는 장소를 제거한다.
5. 예산 조건과 크게 맞지 않는 장소를 제거한다.
6. 남은 후보 중 하나를 서버에서 무작위로 선정한다.
7. 선정된 장소를 데이터베이스에 저장하고 결과를 잠근다.
8. 클라이언트가 새로고침되어도 같은 장소가 유지되어야 한다.

완전히 임의의 좌표를 선정하지 말고,
실제로 방문할 수 있는 등록된 장소만 선정하세요.

조건을 만족하는 장소가 없으면 무리하게 선정하지 말고 다음 메시지를 보여 주세요.

“현재 조건에 맞으면서 영업 중인 장소를 찾지 못했습니다.
이동 시간, 예산 또는 제외 조건을 조금 넓혀 주세요.”

# 4. 영업 중 확인

미스터리 스팟으로 선정하기 직전에 반드시 영업 여부를 다시 확인하세요.

가능하면 장소 API가 제공하는 다음 정보를 사용하세요.

- business status
- current opening hours
- open now
- 다음 영업 종료 시각

Google Places API를 사용할 경우 `businessStatus`,
`currentOpeningHours.openNow`, `nextCloseTime`에 해당하는 정보를 활용하세요.

영업 정보가 없는 장소를 임의로 “영업 중”이라고 표시하면 안 됩니다.

장소 유형에 따라 다음처럼 처리하세요.

- 영업 중으로 확인됨: 후보에 포함
- 영업 종료 또는 폐업: 후보에서 제외
- 영업시간 정보 없음: 원칙적으로 후보에서 제외
- 24시간 운영 또는 운영시간이 없는 공공 야외 공간:
  별도의 장소 유형 규칙으로 처리

단순히 현재 영업 중인지만 확인하지 말고,
예상 이동 시간과 최소 체류 시간을 고려했을 때 도착 전에 문을 닫는 장소도 제외하세요.

예:
- 현재 시각 18:00
- 예상 이동 시간 40분
- 최소 체류 시간 60분
- 영업 종료 19:00

이 경우 방문하기 어려우므로 후보에서 제외합니다.

최종 확인 시각과 상태를 저장하고 화면에 다음처럼 표시하세요.

“영업 여부 확인 완료 · 오후 6:03 기준”

외부 장소 데이터가 실제와 다를 수 있으므로 공개 후에는
장소 공식 페이지 또는 전화번호를 확인할 수 있게 하세요.

# 5. 미스터리 내비게이션

OMYS 자체 추천 모드에서는 목적지 이름, 주소, 사진, 카테고리를
도착 전까지 공개하지 마세요.

지도 화면에는 다음 정보만 보여 주세요.

- 현재 위치
- 이동 경로
- 남은 거리
- 예상 도착 시간
- 진행률
- 현재 이동 방향
- “목적지는 아직 비밀입니다” 문구

목적지 마커, 장소명, 정확한 주소, 주변에서 정답을 쉽게 알 수 있는 정보는
숨기세요.

외부 지도 앱을 그대로 열면 목적지 이름이 노출될 수 있으므로,
가능하면 OMYS 내부 지도에 경로 polyline을 그려 주세요.

진짜 음성 턴바이턴 내비게이션 구현이 MVP 범위를 크게 넘는다면
다음 수준으로 구현해도 됩니다.

- 현재 GPS 위치 표시
- 경로 polyline
- 다음 이동 방향
- 남은 거리
- 예상 도착 시간
- 일정 간격 위치 갱신

사용자가 목적지 반경 100m 안에 들어오면 다음 버튼을 활성화하세요.

“미스터리 스팟 공개하기”

GPS 오차를 고려하여 사용자가 너무 가까이 왔는데도 공개되지 않는 문제를
방지하세요. 위치 권한이 없거나 GPS가 불안정하면 방장 또는 사용자에게
수동 공개 기능을 제공하되, 확인 절차를 한 번 거치세요.

친구 추천 모드에서는 당첨 장소를 제출한 참가자가 가이드 역할을 합니다.
그 참가자에게만 목적지와 길찾기 정보를 보여 주세요.

# 6. 다시 뽑기 규칙

무제한 다시 뽑기는 미스터리 경험을 약하게 만들기 때문에 제한하세요.

기본 규칙:

- 추첨 애니메이션이 끝난 뒤 결과를 서버에 저장하고 잠근다.
- 새로고침하거나 다시 접속해도 결과가 바뀌지 않는다.
- 방장은 출발 전에 최대 1회만 다시 뽑을 수 있다.
- 다시 뽑을 때 확인 모달을 보여 준다.
- 다시 뽑은 뒤에는 이전 장소가 다시 선택되지 않게 한다.
- 두 번째 결과는 더 이상 다시 뽑을 수 없다.
- 출발 버튼을 누른 이후에는 다시 뽑을 수 없다.
- 영업 종료, 임시 휴업처럼 방문 불가능한 상태가 확인되면
  다시 뽑기 횟수를 소모하지 않고 자동으로 대체 장소를 선정한다.

다시 뽑기 허용 여부는 방 생성 시 설정할 수 있게 하세요.

우리가 어디로 가는 중인지 맞혀 보세요.”

# 9. 디자인 방향

디자인 키워드:

- 미스터리
- 설렘
- 즉흥적인 외출
- 친구들과 하는 게임
- 과도하게 어둡거나 공포스럽지 않음

권장 스타일:

- 밝은 배경
- 보라색 또는 블루 계열 포인트 컬러
- 카드 기반 UI
- 위치 핀, 물음표, 봉인된 봉투 등의 시각 요소
- 추첨 시 짧은 애니메이션
- 중요한 버튼은 엄지손가락으로 누르기 쉬운 크기
- 한국어 UI

화려한 기능보다 사용 흐름의 이해 가능성과 모바일 사용성을 우선하세요.

# 11. API와 보안

외부 장소 API 키는 프런트엔드에 직접 노출하지 말고 서버에서 호출하세요.
브라우저용 지도 SDK 키가 필요한 경우 도메인 제한을 적용할 수 있게 안내하세요.

다음을 검증하세요.

- 초대 코드 추측 방지
- 참가자 토큰으로 본인 식별
- 다른 참가자의 비밀 후보 조회 방지
- 서버에서 추첨 실행
- 중복 추첨 요청 방지
- 결과 잠금
- API rate limit 대응
- 외부 API timeout 및 실패 처리
- 사용자 입력 길이 제한
- HTML/script 입력 방지
- CORS 환경변수 설정
- API 키가 Git에 포함되지 않도록 `.env.example` 제공

장소 API 호출 결과는 짧은 시간 캐싱하여 비용과 요청 수를 줄이세요.
하지만 최종 선정 직전 영업 여부는 다시 확인해야 합니다.

# 13. 테스트 조건

다음 시나리오를 반드시 테스트하세요.

1. 방장이 방을 만들고 초대 링크를 복사할 수 있다.
2. 두 명 이상의 사용자가 서로 다른 브라우저에서 같은 방에 입장한다.
3. 각 사용자는 자신의 후보만 볼 수 있다.
4. 모든 사용자가 후보를 제출한 뒤 추첨할 수 있다.
5. 추첨 결과가 새로고침 후에도 유지된다.
6. 허용된 경우에만 1회 다시 뽑을 수 있다.
7. 출발 후에는 다시 뽑을 수 없다.
8. 영업 종료된 장소는 선정되지 않는다.
9. 조건에 맞는 장소가 없으면 안내 메시지가 표시된다.
10. 목적지가 도착 전까지 API 응답과 화면에서 노출되지 않는다.
11. 목적지 반경에 도착하면 공개할 수 있다.
12. 위치 권한을 거부한 경우 대체 흐름이 작동한다.
13. 공유 링크에서 공개 전 목적지가 노출되지 않는다.
14. 모바일 크기에서 주요 화면이 깨지지 않는다.

백엔드 단위 테스트와 핵심 프런트엔드 테스트를 작성하세요.
가능하면 브라우저 기반 E2E 테스트도 작성하세요.
